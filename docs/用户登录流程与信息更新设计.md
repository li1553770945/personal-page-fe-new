**问题1**：我有一个usercontext，一个userlogo，还有login和register的组件弹窗。你认为这些的交互流程是怎么样的？比如点击login之后，登录的函数应该在login组件自己内完成还是统一的地方比如userlogo还是usercontext，登录成功之后由谁来更新用户信息？是usercontext自己更新，还是userlogo更新通过setuser传递给context？

根据最佳实践以及你现有的代码结构（使用 zustand 作为状态管理），最推荐的流程是 "UI 触发 -> Context 处理业务 -> State 更新 -> UI 响应" 。

### 1. 推荐的交互流程 (Flux/Redux 模式)
- UserContext (大脑) :
  - 职责 : 持有 user 数据，提供 login 、 logout 、 refresh 等核心 业务方法 。
  - 逻辑 : 这里是唯一应该修改 user 状态和操作 localStorage (Token) 的地方。
- Login 组件 (手) :
  - 职责 : 收集用户输入 (表单)，处理表单验证。
  - 逻辑 : 用户点击登录后，调用 UserContext.login(data) 。它 不应该 直接关心 Token 存哪里，也不应该自己去 setUser 。
- UserLogo 组件 (脸) :
  - 职责 : 纯展示。根据 UserContext 中的 user 状态显示头像或登录按钮。
  - 逻辑 : 监听 user 变化自动更新视图。不需要手动传递数据。
